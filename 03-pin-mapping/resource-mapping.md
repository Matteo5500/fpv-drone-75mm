# Betaflight Resource Mapping — BETAFPVF411

## What is Resource Mapping

Betaflight uses an abstraction layer between firmware and hardware.
Instead of hardcoding pin assignments, a configurable resource table
maps software functions to physical MCU pins.

This allows the same firmware binary to run on different FC hardware
by changing only the resource configuration.

---

## CLI Resource Command

In Betaflight CLI, the `resource` command reads and writes
the resource mapping:

```
# Read all current resource assignments
resource

# Assign a resource to a different pin
resource MOTOR 1 A08

# Remove a resource assignment
resource MOTOR 1 none
```

Pin notation: `A08` = Port A, Pin 8 = PA8

## Complete Resource Map — BETAFPVF411

### Motors (DSHOT300)

```
resource MOTOR 1 A08   # PA8  → TIM1_CH1 → Motor 1 (front-right, CW)
resource MOTOR 2 B06   # PB6  → TIM4_CH1 → Motor 2 (front-left,  CCW)
resource MOTOR 3 B07   # PB7  → TIM4_CH2 → Motor 3 (rear-right,  CCW)
resource MOTOR 4 B08   # PB8  → TIM4_CH3 → Motor 4 (rear-left,   CW)
```

Motor layout (top view):
```
        FRONT
  M2 (CCW) | M1 (CW)
  ──────────────────
  M3 (CCW) | M4 (CW)
        REAR
```

Motors on diagonal are always same rotation direction.
This balances the reaction torques — without this, the
drone would spin uncontrollably on the yaw axis.

### Serial Ports (UART)

```
resource SERIAL_TX 1 A09   # PA9  → USART1_TX → ELRS RX (CRSF out)
resource SERIAL_RX 1 A10   # PA10 → USART1_RX → ELRS RX (CRSF in)
resource SERIAL_TX 2 A02   # PA2  → USART2_TX → VTX SmartAudio
resource SERIAL_RX 2 A03   # PA3  → USART2_RX → VTX feedback
```

### SPI Bus 1 — IMU

```
resource SPI_SCK  1 A05    # PA5 → SPI1_SCLK → ICM-42688-P clock
resource SPI_MISO 1 A06    # PA6 → SPI1_MISO → ICM-42688-P data to FC
resource SPI_MOSI 1 A07    # PA7 → SPI1_MOSI → ICM-42688-P data from FC
resource GYRO_CS    A04    # PA4 → GPIO_OUT  → ICM-42688-P Chip Select
```

### SPI Bus 2 — Blackbox Flash

```
resource SPI_SCK  2 B13    # PB13 → SPI2_SCLK → Flash clock
resource SPI_MISO 2 B14    # PB14 → SPI2_MISO → Flash data to FC
resource SPI_MOSI 2 B15    # PB15 → SPI2_MOSI → Flash data from FC
resource FLASH_CS   B12    # PB12 → GPIO_OUT  → Flash Chip Select
```

### Sensors and Peripherals

```
resource GYRO_EXTI A01     # PA1 → GPIO_IN (interrupt) → IMU data ready
resource ADC_BATT  A00     # PA0 → ADC1_CH0 → VBAT voltage sensing
resource BEEPER    C15     # PC15 → GPIO_OUT → locator buzzer
```

## Key Concepts

### Hardware Timer for DSHOT
DSHOT signals are generated by hardware timers (TIM1, TIM4),
not by CPU software loops. This guarantees precise timing
independent of CPU load — critical for motor control.

### GYRO_EXTI — Interrupt-Driven Loop
The IMU signals data readiness via the EXTI pin (hardware interrupt).
Betaflight's PID loop is triggered by this interrupt, not by
a software timer. This synchronizes the loop precisely with
the IMU sampling rate (up to 8kHz).

### SPI Bus Isolation
IMU (SPI1) and Flash (SPI2) use separate SPI buses deliberately.
IMU is read at 8kHz; Flash writes happen at lower, irregular
intervals. Sharing a bus would create contention and
unpredictable latency spikes on the gyro read path.

### ADC Voltage Divider
The battery voltage (3.2–4.35V) exceeds the ADC input range
(0–3.3V). A resistor voltage divider on the PCB scales it down.
Betaflight applies a calibration scale factor (`vbat_scale`)
to convert the ADC reading back to the actual battery voltage.
